📋 1. Tổng quan quy trình

Quy trình phát triển phần mềm kết hợp SDD (Software Design Document) và TDD (Test-Driven Development) đảm bảo chất lượng code từ giai đoạn thiết kế đến triển khai. Đặc biệt quan trọng khi chỉnh sửa chức năng có sẵn — nơi rủi ro regression bugs cao nhất.

SDD trả lời câu hỏi "làm NHƯ THẾ NÀO?" — TDD đảm bảo "làm ĐÚNG chưa?" Kết hợp cả hai tạo nên quy trình phát triển chặt chẽ và kiểm chứng được.

📐 2. Quy trình SDD — Software Design Document

SDD là tài liệu mô tả kiến trúc, thiết kế chi tiết và các quyết định kỹ thuật của phần mềm. SDD được tạo sau SRS và trước khi code.

2.2. Quy trình tạo SDD cho chức năng mới

📐 Quy trình SDD: Feature Mới → Chỉnh sửa
  1. Phân tích yêu cầu — Đọc SRS, xác định scope chức năng cần thiết kế
  2. Vẽ wireframe — Tạo wireframe dạng markdown (ASCII layout + component specs)
  3. Định nghĩa data model — Types/Interfaces cho dữ liệu (TypeScript)
  4. Thiết kế component tree — Cây component, props, state management
  5. Viết design doc — Ghi lại tất cả quyết định thiết kế vào SDD
  6. Review — Đội ngũ review SDD trước khi bắt đầu code

🧪 3. Quy trình TDD — Test-Driven Development

TDD là phương pháp phát triển viết test trước, code sau. Mỗi tính năng đều trải qua 3 giai đoạn: RED → GREEN → REFACTOR

🔴
RED
Viết test case mô tả hành vi mong muốn.
Test PHẢI FAIL vì chưa có code.
→
🟢
GREEN
Viết code tối thiểu để test PASS.
Không tối ưu, chỉ cần đúng.
→
🔵
REFACTOR
Cải thiện code không đổi hành vi.
Tất cả test vẫn PASS.

3.1. Ví dụ TDD: Hàm calculateDiscount()

🔴 Bước 1: RED — Viết test trước
// __tests__/utils.test.ts test('returns correct discount %', () => { expect(calculateDiscount(100000, 80000)) .toBe(20); // Mong đợi 20% }); test('returns 0 when original is 0', () => { expect(calculateDiscount(0, 80000)) .toBe(0); }); // ❌ Chạy test → FAIL (chưa có hàm) FAIL 2 tests failed
🟢 Bước 2: GREEN — Code vừa đủ
// src/lib/utils.ts export function calculateDiscount( original: number, current: number ): number { if (original <= 0) return 0; return Math.round( ((original - current) / original) * 100 ); } // ✅ Chạy test → PASS PASS 2 tests passed

🔧 4. Quy trình chỉnh sửa chức năng (quan trọng nhất)

⚠️
80% bugs trong phần mềm đến từ việc chỉnh sửa code hiện có — không phải từ code mới. Quy trình SDD+TDD đặc biệt quan trọng khi sửa chức năng!

Khi cần chỉnh sửa một chức năng đã có (ví dụ: thay đổi logic giỏ hàng, thêm field mới cho sản phẩm, sửa thuật toán tìm kiếm), quy trình phải tuân thủ 6 bước sau:

🔧 Quy trình chỉnh sửa chức năng — 6 bước
  1. Cập nhật SRS/SDD — Ghi nhận yêu cầu thay đổi vào tài liệu. Mô tả: hành vi cũ → hành vi mới. Cập nhật wireframe nếu ảnh hưởng UI.
    // Ví dụ: Thay đổi discount code từ 4 mã → 6 mã // SRS ghi: // - Cũ: CYBER10(10%), CYBER20(20%), NEWUSER(15%), FLASHSALE(25%) // - Mới: + SUMMER30(30%), VIP50(50%)
  2. 🔴 RED: Viết test cho hành vi MỚI — Test case cho cả hành vi mới VÀ đảm bảo hành vi cũ không bị phá vỡ.
    // Thêm test mới (phải FAIL) test('applies SUMMER30 code for 30%', () => { act(() => { cart.addItem(product); }); // 5,000,000đ act(() => { cart.applyDiscount('SUMMER30'); }); expect(cart.discountPercent).toBe(30); expect(cart.total).toBe(3500000); // 5M × 0.7 }); test('applies VIP50 code for 50%', () => { act(() => { cart.applyDiscount('VIP50'); }); expect(cart.discountPercent).toBe(50); }); // Chạy test: PASS 17 tests (cũ vẫn pass) FAIL 2 tests (mới fail — đúng mong đợi)
  3. 🟢 GREEN: Sửa code để pass — Chỉnh sửa source code tối thiểu để test mới PASS, đồng thời test cũ vẫn PASS.
    // src/context/CartContext.tsx — Thêm 2 mã mới const VALID_CODES: Record<string, number> = { CYBER10: 10, CYBER20: 20, NEWUSER: 15, FLASHSALE: 25, SUMMER30: 30, // ← THÊM MỚI VIP50: 50, // ← THÊM MỚI }; // Chạy lại test: PASS 19 tests (tất cả pass ✅)
  4. 🔵 REFACTOR: Tối ưu (nếu cần) — Cải thiện code quality mà không thay đổi hành vi. Chạy lại toàn bộ test → vẫn PASS.
  5. E2E Test — Cập nhật/thêm E2E test cho luồng hoàn chỉnh. Kiểm tra từ góc nhìn người dùng.
  6. ✅ Kiểm tra Coverage — Chạy npm run test:coverage để đảm bảo code mới được bao phủ ≥ 90%. Tạo báo cáo.
✅
Nguyên tắc vàng: Nếu test cũ vẫn PASS sau khi sửa code → không có regression bug. Nếu test cũ FAIL → phát hiện lỗi ngay lập tức, trước khi deploy.

🎯 7. Lợi ích khi áp dụng SDD + TDD

🛡️ Phòng ngừa Regression
Test cũ tự động phát hiện khi code mới phá vỡ tính năng cũ — giảm 80% bugs khi sửa code.
📄 Tài liệu sống
Test cases chính là tài liệu mô tả hành vi hệ thống — luôn cập nhật theo code.
⚡ Tự tin Deploy
Coverage ≥ 90% + tất cả test PASS → tự tin deploy mà không lo lỗi ẩn.
🔄 Refactor an toàn
Có test bảo vệ, developer tự tin refactor code mà không sợ phá vỡ logic.
👥 Teamwork hiệu quả
SDD giúp team hiểu thiết kế trước khi code. TDD đảm bảo code review dễ dàng.
📈 Đo lường được
Coverage C0/C1, test pass rate, execution time — tất cả đều định lượng được.
